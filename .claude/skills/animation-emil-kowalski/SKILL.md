# Animation Guidelines — Emil Kowalski

Источник: https://emilkowal.ski/ui/great-animations

Задача: анимация должна ощущаться естественно и не мешать восприятию интерфейса.

## Принципы

### 1. Естественность
- Предпочитай spring-анимации линейным/ease-in-out там, где движение имитирует физический объект (drag, modal, drawer).
- Избегай мгновенных скачков состояния без перехода — они читаются как баг, не как дизайн.

### 2. Скорость и отзывчивость
- Длительность < 300ms для большинства UI-переходов (hover, toggle, tooltip).
- `ease-out` — стандарт для появления элементов (кажется быстрее, чем `ease-in-out`).
- Более длинные анимации (400–600ms) — только для крупных layout-переходов (page transition, модалки на весь экран).

### 3. Целесообразность
- Анимация — только если несёт информационную ценность (показывает связь, направление, причину/следствие).
- Не анимировать часто повторяемые действия (ввод текста, частые клики, keyboard-driven операции) — раздражает и тормозит работу.

### 4. Производительность
- Анимировать только `transform` и `opacity` — не требуют пересчёта layout/paint.
- Избегать анимации `width`, `height`, `top`, `left`, `box-shadow` без crossfade-трюка.
- Использовать аппаратное ускорение: CSS transitions/animations или Web Animations API поверх requestAnimationFrame-хаков.

### 5. Прерываемость (interruptibility)
- Анимация должна корректно подхватывать новое состояние при повторном действии пользователя до завершения текущей (не блокировать очередью, не дёргаться).
- Пример: пользователь быстро закрывает и открывает модалку — анимация должна плавно развернуться в обратную сторону, а не завершить старую и запустить новую с рывком.

### 6. Доступность
- Уважать `prefers-reduced-motion`: отключать/упрощать неважные декоративные анимации, оставлять только функциональные (индикация состояния) в сокращённом виде.
- Не использовать анимацию как единственный носитель критичной информации.

### 7. Целостность
- Timing (duration/easing) — единый токен/пресет на проект, не разнобой значений в каждом компоненте.
- Соответствие анимации общей эстетике: резкий brutalist-интерфейс — резкие easing; мягкий продукт — spring.

## Чеклист ревью анимации

- [ ] Использует `transform`/`opacity`, не layout-свойства?
- [ ] Длительность соответствует масштабу изменения (micro <300ms, layout 400–600ms)?
- [ ] Есть `prefers-reduced-motion` fallback?
- [ ] Анимация прерываема без визуального рывка?
- [ ] Несёт смысл (направление/причину), не просто "красиво"?
- [ ] Easing/duration взяты из общего пресета проекта, не magic number?

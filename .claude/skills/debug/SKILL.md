# debug

Дисциплинированный workflow для сложных багов. Не пропускай фазы без явного обоснования.

---

## Фаза 1 — Feedback Loop

**Это весь навык. Остальное — механика.**

Если есть fast, deterministic, agent-runnable pass/fail сигнал — баг найдётся. Без него никакое чтение кода не поможет.

Трать здесь непропорционально много усилий. Будь агрессивен. Не сдавайся.

### Способы построить loop — пробуй в этом порядке

1. **Failing test** на нужном seam — unit, integration, e2e.
2. **curl / HTTP скрипт** против запущенного dev-сервера.
3. **CLI invocation** с fixture-input, diff stdout против known-good snapshot.
4. **Headless browser** (Playwright) — кликает UI, проверяет DOM/консоль/сеть.
5. **Replay captured trace** — сохрани реальный запрос/payload/event log, прогони изолированно.
6. **Throwaway harness** — минимальный subset системы (один сервис, моки), одним вызовом функции.
7. **Property / fuzz loop** — если "иногда неверный output": 1000 случайных входов.
8. **Bisection harness** — если баг появился между двумя состояниями: автоматизируй `git bisect run`.
9. **Differential loop** — один input через старую vs новую версию, diff outputs.
10. **HITL bash script** — крайний случай: если нужен человек, структурируй его действия скриптом.

### Итерация над loop

- Можно ускорить? (кэш setup, сузить scope)
- Сигнал точнее? (assert на симптом, не на "не упало")
- Детерминированнее? (зафиксируй время, seed RNG, изолируй FS, заморозь сеть)

30-секундный flaky loop почти бесполезен. 2-секундный детерминированный — суперсила.

### Недетерминированные баги

Цель — не чистый repro, а **высокий rate воспроизведения**. Запускай 100×, добавь стресс, суживай timing window. 50%-flake — дебажится; 1% — нет. Поднимай rate до debuggable.

### Если loop построить не удаётся

Стоп. Скажи явно. Перечисли что пробовал. Попроси у пользователя: (а) доступ к окружению, (б) captured artifact (HAR, log dump, screen recording), (в) разрешение на временную prod-инструментацию. **Не переходи к Hypothesise без loop.**

## Фаза 2 — Reproduce

Запусти loop. Убедись:

- [ ] Баг совпадает с тем, что описал пользователь — не соседний баг.
- [ ] Воспроизводится стабильно (или с достаточным rate).
- [ ] Зафиксирован точный симптом (сообщение ошибки, неверный output, timing).

Не переходи дальше без воспроизведения.

---

## Фаза 3 — Hypothesise

Сформулируй **3–5 ranked гипотез** до тестирования любой из них.

Каждая гипотеза — **фальсифицируемая**:

> "Если X является причиной, то при изменении Y баг исчезнет / при изменении Z станет хуже."

Если не можешь сформулировать предсказание — это не гипотеза, а vibe. Отбрось или уточни.

**Покажи список пользователю до тестирования** — у него может быть domain knowledge, которое сразу переранжирует. Если пользователь недоступен — продолжай по своему ранжированию.

**Выход:** список гипотез подтверждён / пользователь переранжировал.

---

## Фаза 4 — Instrument

Каждый probe → конкретная гипотеза из Фазы 3. **Меняй одну переменную за раз.**

Приоритет инструментов:
1. **Debugger / REPL** — один breakpoint лучше десяти логов.
2. **Точечные логи** на границах, разделяющих гипотезы.
3. Никогда "log everything and grep".

**Тегируй все debug-логи** уникальным префиксом: `[DEBUG-xxxx]` (4 chars, per-session, напр. `[DEBUG-a4f2]`). Cleanup = один grep. Нетегированные логи выживают; тегированные — умирают.

**Perf regression:** логи бесполезны. Строй timing harness, запускай profiler, смотри query plan. Измеряй сначала, фикси потом.

**Выход:** одна гипотеза подтверждена или опровергнута. Если ни одна — вернись к Hypothesise с новыми данными.

---

## Фаза 5 — Fix + Regression Test

Пиши regression test **до фикса** — но только при наличии **correct seam**.

Correct seam: тест воспроизводит реальный баг-паттерн так, как он возникает на call site. Если seam слишком мелкий (unit-тест не может воспроизвести chain) — тест даёт ложную уверенность.

**Нет correct seam = находка.** Зафикси это: архитектура мешает закрепить баг тестом. Передай в post-mortem.

При наличии seam:
1. Превращай minimised repro в failing test.
2. Смотри, как он падает.
3. Применяй фикс.
4. Смотри, как он проходит.
5. Запускай Phase 1 loop против оригинального (не minimised) сценария.

**Выход:** Phase 1 loop проходит на оригинальном сценарии.

---

## Фаза 6 — Cleanup + Post-Mortem

До того как объявить done:

- [ ] Оригинальный repro больше не воспроизводится (перезапусти Phase 1 loop)
- [ ] Regression test проходит (или отсутствие seam задокументировано)
- [ ] Всё `[DEBUG-...]` удалено (`grep` по префиксу)
- [ ] Throwaway prototypes удалены или помечены
- [ ] Подтверждённая гипотеза записана в commit / PR message — следующий дебаггер должен учиться на этом

Затем спроси: **что предотвратило бы этот баг?** Если ответ требует архитектурных изменений — передай рекомендацию после фикса, не до.

# Политика безопасности

## Поддерживаемые версии

Поддерживается актуальная версия kit (см. [`.claude/kit-version`](.claude/kit-version)).

## Как сообщить об уязвимости

**Не открывай публичный Issue** — уязвимости разбираются приватно.

Используй встроенный механизм GitHub: **Security → Report a vulnerability** в этом репозитории. Укажи:

- затронутый файл / хук / skill,
- шаги воспроизведения,
- потенциальное влияние.

Ответ — в течение 72 часов. Публичное раскрытие — после исправления, по согласованию с репортером.

## Затронутые компоненты

На что обращаем особое внимание:

- **Хуки** (`.claude/hooks/`, `.cursor/hooks/`) выполняются как код в среде пользователя — любая инъекция команды является уязвимостью.
- **`safety-guard`** блокирует деструктивные команды (`DROP TABLE/DATABASE`, `git push --force` на protected-ветках) — обход этой защиты считается уязвимостью.
- **Секреты**: токены хранятся в `*/secrets/github_token` и `*.env`, исключённых через [`.gitignore`](.gitignore). Случайный коммит секрета — сообщи по каналу выше.

## Рекомендации

- Не вставляй реальный PAT в файлы вне `secrets/`.
- Проверяй `.gitignore` перед первым коммитом в целевом проекте.
- Минимальный scope GitHub-токена: `contents:read`.

## Вендоринг стороннего skill

Skill — не текст, а инструкции, которые агент исполняет **с правами пользователя**. Аудиты находят prompt injection примерно в трети публичных skills. Поэтому любой skill из внешнего источника проходит приёмку — иначе он не попадает в кит.

### Чеклист

1. **Прочитай SKILL.md целиком.** Не по диагонали. Ищи: инструкции, обращённые не к задаче, а к агенту («ignore previous», «do not tell the user», «before anything else, run…»); обращения к секретам, `.env`, ключам; сетевые вызовы, не связанные с назначением скилла.
2. **Прочитай каждый файл в `scripts/`.** Это код, который выполнится. Минифицированный бандл без исходника — повод отказаться либо собрать самому из исходников.
3. **Зафиксируй провенанс.** `source` + `sourceType` + `sourceRef` (коммит, не `main`). Ветка — не пин: перевендоришь и получишь что угодно.
4. **Запиши в `skills-lock.json`** — `computedHash` для `SKILL.md` и `fileHashes` для **каждого остального файла скилла**: и `scripts/` (код, который выполнится), и `references/` (инструкции, которым агент подчинится). Оба доходят до модели с правами пользователя, поэтому пинятся оба. Без записи в локе CI упадёт (fail closed), и это правильно.
5. **Задокументируй каждую kit-модификацию** в `note`. Незадокументированная правка вендоренного файла неотличима от подмены.
6. **Заданное направление / узкая ниша → `disable-model-invocation: true`.** Скилл, который модель не должна выбирать сама, не участвует в автовыборе.

### При обновлении вендоренного skill

Хеш разошёлся — это **не** повод «просто обновить хеш». Порядок такой:

```
git diff <старый файл> <новый файл>   # прочитай diff целиком
```

Убедись, что изменения соответствуют заявленному апстримом, и только потом обновляй `computedHash` / `fileHashes` и `sourceRef`. Хеш, обновлённый не глядя, — это ровно тот rug pull, от которого лок и защищает.

### Границы того, что покрывает CI

`check-skills.js` ловит **изменение** запиненного скилла и **появление** незапиненного. Он не ловит вредоносный код, запиненный с самого начала. `node --check` и `ruff` проверяют синтаксис, а не намерение. Единственный барьер против злонамеренного скилла на входе — пункты 1 и 2 выше, выполненные человеком.

## Границы `permissions` — что они НЕ закрывают

В `settings-template.json` закрыты `Read(./.env)`, `Read(./**/secrets/**)`, запись в `.env` (`Edit`/`Write`) и `mcp__playwright__browser_run_code_unsafe`.

Это правила для **инструментов**, а не для файловой системы. Запрет `Read(./**/secrets/**)` останавливает инструмент `Read` — но **не** `cat` через Bash. Сам кит читает `github_token` именно так (`cursor-kit-maintenance`, проверка версии), поэтому запрет на уровне Bash сломал бы кит и здесь не ставится.

Что это значит на практике:

- Правило `Read(...)` — защита от **случайного** чтения секрета агентом и повод для явного отказа, а не барьер против целенаправленного обхода.
- Барьер на уровне команд — это `safety-guard.js` и (сильнее) `dcg` из `install.sh --with-tools dcg`.
- Единственная надёжная мера — **секретов нет в репозитории**: `.gitignore` покрывает `*/secrets/` и `*.env`, а скиллы записывают только *где найти* секрет, не сам секрет.

Не считай `permissions` защитой от скомпрометированного скилла — считай гигиеной, снижающей площадь случайной утечки.
